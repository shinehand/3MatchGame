# Task Brief

## 작업명

- 100개 스테이지 확장 기획 및 구현 준비

## 목표

- 현재 프로토타입을 100개 스테이지까지 확장 가능한 구조로 정리하고, 난이도가 순차적으로 상승하는 퍼즐 캠페인 설계를 확정한다.

## 범위

- 포함
- 100개 스테이지 기획 곡선
- 스테이지별 맵 구조와 목표 유형 정의
- 100개 스테이지를 담을 데이터 구조와 검증 방식 정의
- 장기 스테이지 진행에 맞는 시각 진행 구조 정의
- 제외
- 100개 스테이지 전부를 이번 턴에 코드로 완전 구현
- 신규 장애물과 신규 특수 블록의 실제 코드 구현
- 스토어 메타, 수익화, 라이브옵스 설계

## 기획 입력

- 핵심 루프
- 동물 블록을 스와이프해 매치하고, 제거 후 낙하와 리필로 연쇄를 만든다.
- 성공 조건
- 스테이지별 목표 달성 후 다음 스테이지로 진행한다.
- 실패 조건
- 남은 이동 수가 0이 되기 전에 목표를 달성하지 못하면 실패한다.
- 난이도 또는 밸런스 메모
- Stage 1~6의 현재 규칙을 출발점으로 삼고, 100개 스테이지 동안 난이도 상승 곡선이 명확해야 한다.

## 아트 입력

- 필요한 화면
- 홈 화면, 스테이지 선택, Stage 시작, 플레이 HUD, 클리어/실패 팝업
- 필요한 에셋
- 배경 세트, 목표 칩 변형, 장애물/특수 블록 이펙트, 난이도 구간별 시각 테마
- UI 제약
- landscape 1920x1080, block-first, 귀여운 동물 얼굴 스티커형 톤 유지
- 임시 리소스 허용 여부
- 허용. 다만 실제 동물 블록 스타일과 톤이 어긋나면 안 된다.

## 개발 입력

- 대상 씬/스크립트
- `scripts/stage_catalog.gd`, 향후 분리될 stage data loader, stage select 관련 씬/스크립트
- 입력 액션
- 탭, 스와이프, 스테이지 선택
- 저장 데이터
- 현재 진행 스테이지, 별 점수, 잠금 해제 상태
- 테스트 방법
- 헤드리스 실행, 스테이지 데이터 유효성 검증, Android 디버그 빌드 설치

## 완료 조건

- 사용자가 확인할 수 있는 결과
- 100개 스테이지 캠페인의 난이도 구조와 제작 기준이 문서로 정리된다.
- 개발자가 검증할 수 있는 결과
- 스테이지 데이터 구조와 검증 방식이 문서화되고, 실제 구현 작업 단위가 분리된다.

## 리스크

- 일정
- 100개 스테이지를 수작업으로 유지하면 수정 비용이 커진다.
- 성능
- 데이터 로딩과 검증을 하드코딩으로만 유지하면 관리 비용이 증가한다.
- 아트 의존성
- 배경과 HUD 변형이 블록 스타일과 어긋날 위험이 있다.
- 기획 미정 요소
- 신규 장애물과 중후반 규칙 확장이 언제 잠금 해제될지 확정이 필요하다.
