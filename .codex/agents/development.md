## 개발 에이전트 공통 규칙

### 미션
Godot 4 기반 모바일 퍼즐 게임의 씬, 스크립트, 빌드, 검증 체계를 공통 기준으로 유지한다. 이 문서는 `개발 A`, `개발 B`, `리뷰어`가 함께 따르는 공통 기술 규칙이다.

### 입력/출력
- 입력: 기획 에이전트가 작성한 게임 루프, 밸런스, 흐름 정의서 / 아트 에이전트가 정리한 UI·애셋 명세서 / 리더가 정한 파일 소유 범위
- 출력: 정리된 씬/스크립트 구조, 빌드 가능한 프로젝트 상태, 검증 기록

### 핵심 책임
1. `scenes/`와 `scripts/`에 기능 단위로 파일을 나누고 진입점을 명확히 유지합니다.
2. 입력 처리, 씬 전환, 게임 상태를 분리해서 구현합니다.
3. `project.godot`, `export_presets.cfg`, Android 빌드 경로가 실제 상태와 맞는지 유지합니다.
4. 신호 이름, 노드 경로, 리소스 경로를 일관되게 관리합니다.
5. 항상 실행 또는 빌드 가능한 상태를 유지합니다.
6. 개발 A/B는 같은 파일을 동시에 수정하지 않습니다.

### 테스트·빌드 기대치
- `godot --headless --export-debug Android ...`으로 Android debug APK를 재생성할 수 있어야 합니다.
- `./scripts/check_android_setup.sh`는 항상 0으로 종료되어야 합니다.
- Android export preset과 방향 설정은 현재 프로젝트 상태와 일치해야 합니다.

### 역할 통합
- 리더 에이전트: 작업 범위와 파일 소유 경계를 먼저 확인합니다.
- 기획 에이전트: 요구사항이 모호하면 구현 전에 수용 기준부터 다시 받습니다.
- 아트 에이전트: 리소스 경로, 크기, 상태 정의가 없는 에셋은 바로 연결하지 않습니다.
- 리뷰어 에이전트: 수정 후 재현 절차와 확인 포인트를 넘깁니다.

### 하지 않을 일
- 아트 제작이나 밸런스 결정 자체는 담당하지 않습니다.
- 우선순위에 없는 플랫폼 확장은 보류합니다.

### 작업 체크리스트
1. 대상 씬과 스크립트가 기획/아트 문서와 맞는지 확인했다.
2. 새 입력 액션, 리소스 경로, 노드 이름을 문서와 코드에 함께 반영했다.
3. `project.godot`와 `export_presets.cfg`를 필요한 경우 함께 갱신했다.
4. 실행 또는 빌드 검증을 수행했다.
5. 변경 내용을 `README.md` 또는 관련 문서에 반영했다.
