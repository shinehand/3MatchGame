# 범용 팀 구조 템플릿

## 목적

이 문서는 특정 프로젝트나 특정 기술 스택에 종속되지 않는 범용 팀 구조 템플릿이다. 새 프로젝트를 시작할 때 그대로 복사해서 팀 역할, 팀장, 서브 에이전트, 핸드오프 방식을 빠르게 정할 수 있다.

## 기본 팀

1. `PM Team`
2. `Planning Team`
3. `Art Team`
4. `Development Team`
5. `QA Team`

## 공통 구조

- 각 팀은 `팀장 에이전트 1명`과 `서브 에이전트 2~5명`으로 시작한다.
- 팀장은 방향 설정, 우선순위, 승인 기준, 핸드오프를 관리한다.
- 서브 에이전트는 좁은 작업 단위를 맡아 빠르게 실행한다.
- 팀장끼리는 의사결정과 충돌 해결을 담당하고, 팀원끼리는 산출물 생산을 담당한다.

## 팀 간 흐름

1. `PM Team`이 목표, 범위, 우선순위, 승인 기준을 정의한다.
2. `Planning Team`이 요구사항과 세부 명세를 만든다.
3. `Art Team`이 UI/비주얼/리소스 기준을 만든다.
4. `Development Team`이 구현하고 내부 리뷰까지 끝낸다.
5. `QA Team`이 최종 사용자 관점으로 검증한다.
6. `PM Team`이 승인하고 다음 작업으로 넘긴다.

## 기본 서브 에이전트 예시

### PM Team

- 일정/우선순위 담당
- 리스크/의존성 담당
- 산출물 통합 담당

### Planning Team

- 시스템 기획 담당
- 사용자 흐름 담당
- 밸런스/운영 기획 담당
- 스토리/월드 기획 담당

### Art Team

- UI/UX 담당
- 아이콘/에셋 명세 담당
- 배경/월드 방향 담당
- 피드백/이펙트 담당

### Development Team

- 코어 로직 담당
- UI/프론트 담당
- 데이터/저장 담당
- 빌드/도구 담당
- 코드 리뷰 담당

### QA Team

- 기능 QA 담당
- UX QA 담당
- 기기/환경 QA 담당
- 회귀 QA 담당

## 운영 원칙

- 개발팀의 코드 리뷰는 같은 팀 안에서 빠르게 돌린다.
- QA팀은 개발팀과 독립적으로 최종 품질을 본다.
- 같은 파일이나 같은 데이터 모델을 동시에 수정해야 하면 소유 범위를 다시 쪼갠다.
- 팀 구조는 프로젝트 규모에 따라 줄이거나 늘릴 수 있지만, `PM -> Planning/Art -> Development -> QA` 흐름은 유지하는 것이 좋다.
