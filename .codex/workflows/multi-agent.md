# Generic Multi-Team Workflow

## 목표

Codex가 한 번의 세션에서도 `PM팀`, `기획팀`, `아트팀`, `개발팀`, `QA팀`을 병렬로 다루되, 산출물 충돌 없이 어떤 프로젝트에도 적용 가능한 방식으로 작업을 진행하도록 한다.

## 권장 흐름

1. 작업 시작 전에 `task-brief`를 만든다.
2. `PM Team`이 목표, 범위, 우선순위, 승인 기준, 담당 배분을 정리한다.
3. `Planning Team`이 요구사항, 완료 조건, 제외 범위를 정리한다.
4. `Art Team`이 화면 구성, 리소스 목록, UI/UX 기준을 정리한다.
5. `Development Team`이 구현, 내부 리뷰, 수정 루프를 진행한다.
6. `QA Team`이 최종 검증과 승인/반려 판단을 진행한다.
7. `PM Team`이 결과를 통합한 뒤 다음 작업으로 넘긴다.

## 핸드오프 규칙

- PM -> All
  필수 전달: 작업 목표, 우선순위, 제외 범위, 승인 기준, 담당 팀
- Planning -> Art
  필수 전달: 사용자 목표, 핵심 흐름, 정보 우선순위, 기능 범위
- Planning -> Development
  필수 전달: 수용 기준, 성공 조건, 실패 조건, 비기능 요구사항
- Art -> Development
  필수 전달: UI 구조, 스타일 방향, 필요한 리소스 목록, 임시 대체 가능 범위
- Development -> QA
  필수 전달: 변경 파일, 구현 범위, 검증 절차, 확인 포인트
- QA -> PM
  필수 전달: 승인 여부, 주요 오류, 차단 이슈, 재작업 우선순위

## 작업 쪼개기 기준

- 1개 에이전트가 하루 이상 걸릴 내용이면 더 작은 단위로 분해한다.
- 산출물이 문서 중심이면 `PM`, `Planning`, `Art`로 먼저 보낸다.
- 산출물이 코드 중심이면 `Development`로 보내되, 앞단 요구사항 문서가 없으면 먼저 되돌린다.
- 같은 파일이나 같은 산출물을 동시에 수정해야 하면 먼저 작업 단위를 다시 쪼갠다.

## 완료 정의

- PM: 역할 분배와 승인 조건이 명시되어 있다.
- 기획: 목표와 수용 기준이 명시되어 있다.
- 아트: 구현 가능한 수준의 리소스 요구사항이 있다.
- 개발: 내부 리뷰까지 끝난 구현과 검증 기록이 있다.
- QA: 최종 승인 또는 반려 근거가 남아 있다.
- 통합: 다음 작업자가 읽고 바로 이어서 일할 수 있다.
