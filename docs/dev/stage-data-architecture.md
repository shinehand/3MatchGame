# Stage Data Architecture

## 목적

현재 `scripts/stage_catalog.gd`의 하드코딩된 6개 스테이지를, 100개 스테이지까지 안정적으로 확장 가능한 데이터 구조로 바꾼다.

핵심 목표는 다음과 같다.

- 스테이지 추가가 코드 수정이 아니라 데이터 수정으로 끝나야 한다.
- 스테이지 순서와 난이도 곡선이 1에서 100까지 일관되게 이어져야 한다.
- 홈 화면, 스테이지 선택, 저장, 검증이 같은 스테이지 ID 체계를 공유해야 한다.
- 보드 모양, 장애물, 특수 규칙, 튜토리얼을 스테이지 단위로 점진 확장할 수 있어야 한다.

## 권장 방향

런타임에서는 기존처럼 `Dictionary` 기반 구조를 유지하되, 스테이지의 원본 저장소는 `GDScript` 하드코딩이 아니라 `JSON` 데이터로 옮긴다.

이유는 다음과 같다.

- 현재 `gameplay.gd`는 스테이지를 `Dictionary`로 읽고 있어서, 런타임 자료형을 유지하면 이행 비용이 작다.
- 100개 스테이지는 수동 GDScript 편집보다 JSON 분할 관리가 낫다.
- JSON은 diff가 읽기 쉽고, 외부 검증 도구와 연결하기 쉽다.
- 이후 필요하면 JSON에서 `Resource`로 변환하는 단계적 확장도 가능하다.

권장하지 않는 방향은 다음과 같다.

- 모든 스테이지를 계속 `scripts/stage_catalog.gd`에 직접 적는 방식
- 지금 단계에서 곧바로 per-stage `Resource` 수백 개를 만드는 방식

`Resource`는 나중에 에디터 기반 편집이 필요해졌을 때 2차 선택지로 둔다.

## 데이터 형태

스테이지는 `id` 기준의 안정적인 레코드로 정의한다. 화면상 순서와 저장 순서는 `id`를 기준으로 고정한다.

권장 스키마 예시는 아래와 같다.

```json
{
  "id": 17,
  "name": "Stage 17",
  "band": "11-20",
  "difficulty": "Normal",
  "theme_key": "meadow_2",
  "moves": 15,
  "board_mask": [
    "00111100",
    "01111110",
    "11111111",
    "11111111",
    "11111111",
    "11111111",
    "01111110",
    "00111100"
  ],
  "targets": {
    "collect": {
      "rabbit": 8,
      "cat": 8
    },
    "score": 1200,
    "blockers": 4
  },
  "spawn_profile": {
    "pool": ["rabbit", "bear", "cat", "chick", "frog"],
    "weights": {
      "rabbit": 3,
      "bear": 2,
      "cat": 3,
      "chick": 2,
      "frog": 1
    }
  },
  "blockers": [
    {"kind": "bush", "row": 2, "col": 2, "hp": 1}
  ],
  "mechanics": {
    "enabled": ["row_special", "col_special", "bomb_special"]
  },
  "tutorial": "..."
}
```

필수 필드는 `id`, `name`, `difficulty`, `moves`, `board_mask`, `targets`, `spawn_profile`, `blockers`, `tutorial`이다.

## 보드 마스크

보드 모양은 `8x8` 고정 전제에서 `board_mask`로 정의한다.

- 한 행은 길이 8의 문자열로 둔다.
- `1`은 활성 칸, `0`은 비활성 칸이다.
- 비활성 칸은 보드 데이터와 렌더링 모두에서 빈 슬롯으로 취급한다.

이 방식의 장점은 다음과 같다.

- 사람이 읽기 쉽다.
- 스테이지별 실루엣 차이를 빠르게 만들 수 있다.
- 현재 낙하 및 리필 로직과도 맞물린다.

추가 확장 여지는 있지만, 100스테이지 규모에서는 우선 문자열 마스크가 가장 단순하다. RLE, 좌표 리스트, 다각형 표현은 검증 비용이 더 커서 나중 단계로 미룬다.

## 장애물 인코딩

현재의 `blockers`는 `row`, `col` 중심의 단순 배치지만, 100스테이지에서는 다음 형태로 확장 가능한 구조가 필요하다.

- `kind`: `bush`, `ice`, `lock`, `crate` 같은 장애물 유형
- `row`, `col`: 위치
- `hp`: 제거에 필요한 충격 수
- `tags`: 선택적 동작 플래그

권장 원칙은 다음과 같다.

- 지금 구현된 덤불은 `kind: bush`, `hp: 1`로 유지한다.
- 미래 장애물은 동일 배열에 누적하되, 게임 로직은 `kind`와 `hp`만 읽어 동작하도록 만든다.
- 장애물은 반드시 활성 칸 위에만 놓인다.

이렇게 해 두면 장애물 종류가 늘어나도 스테이지 데이터 구조는 바뀌지 않는다.

## 확장성

100개 스테이지는 단순 난이도 상승뿐 아니라, 기능 잠금 해제와 규칙 변형을 담을 수 있어야 한다.

권장 확장 필드는 다음과 같다.

- `theme_key`: 배경과 UI 톤을 연결하는 키
- `mechanics.enabled`: 이 스테이지에서 사용 가능한 규칙 목록
- `tutorial`: 플레이어 안내 문구 또는 튜토리얼 단계 정의
- `tags`: `collect_focus`, `score_focus`, `obstacle_focus` 같은 분류용 태그
- `move_budget_band`: 보정용 난이도 구간

중요한 원칙은 스테이지 데이터가 기능을 직접 구현하지 않고, 기능의 사용 여부만 선언해야 한다는 점이다. 실제 로직은 `gameplay.gd` 또는 별도 규칙 모듈에 남겨 둔다.

## 스테이지 선택과 저장

스테이지 데이터는 저장 시스템과 분리하지 말고, 같은 `id` 기준으로 연결한다.

권장 저장 데이터는 다음과 같다.

- `highest_unlocked_stage_id`
- `cleared_stage_ids`
- `best_score_by_stage_id`
- `best_star_by_stage_id`

여기서 중요한 점은 배열 인덱스가 아니라 `id`를 저장 기준으로 삼는 것이다.

이유는 다음과 같다.

- 중간에 스테이지를 추가하거나 순서를 조정해도 저장 데이터가 깨지지 않는다.
- 홈 화면과 스테이지 선택 화면이 같은 식별 체계를 공유한다.
- 100개 스테이지 중 일부를 이벤트 스테이지로 교체해도 저장 호환성이 유지된다.

## 검증 절차

100개 스테이지 체계에서는 수동 확인만으로는 부족하므로, 데이터 검증 단계를 별도로 둔다.

권장 검증 항목은 다음과 같다.

- `id`는 1부터 100까지 중복 없이 연속되어야 한다.
- `board_mask`는 항상 8행 8열이어야 한다.
- 활성 칸이 최소 1칸 이상 있어야 한다.
- 모든 `blockers`는 활성 칸 위에 있어야 한다.
- 알려진 동물 ID만 스폰 풀에 들어가야 한다.
- `moves`, `targets`, `spawn_profile`은 음수나 비정상 값을 가질 수 없다.
- 난이도 구간별 이동 수와 목표량이 역전되지 않아야 한다.

검증 단계는 두 겹이 필요하다.

1. 정적 검증
   - JSON 파싱
   - 필수 키 확인
   - 범위 확인
   - 마스크/장애물 좌표 검증
2. 실행 검증
   - 헤드리스 Godot에서 스테이지를 로드
   - 보드 생성 성공 여부 확인
   - 유효한 이동이 생성되는지 확인
   - 장애물/마스크 때문에 보드가 깨지지 않는지 확인

장기적으로는 더 강한 검증으로 넘어갈 수 있지만, 100스테이지 첫 버전에서는 위 두 단계면 충분하다.

## 마이그레이션 단계

`scripts/stage_catalog.gd`에서 바로 100개로 늘리지 않고, 아래 순서로 옮긴다.

1. 현재 6개 스테이지를 기준 데이터로 동결한다.
2. 스테이지 스키마를 JSON으로 정한다.
3. `data/stages/` 아래에 band 단위 파일 또는 단일 카탈로그 파일을 둔다.
4. 기존 `stage_catalog.gd`는 JSON 로더 래퍼로 바꾼다.
5. gameplay와 선택 화면은 계속 `Array[Dictionary]` 인터페이스를 쓴다.
6. 저장 시스템은 `stage_id` 기반으로 먼저 붙인다.
7. 검증 스크립트를 추가해 100개 스테이지 전부를 체크한다.
8. 검증이 안정되면 `stage_catalog.gd`의 하드코딩 배열을 제거한다.

## 권장 파일 구조

- `data/stages/stages_001_010.json`
- `data/stages/stages_011_020.json`
- ...
- `data/stages/stages_091_100.json`
- `scripts/stage_catalog.gd`
- `scripts/stage_data_validator.gd`
- `docs/game/stage-production-guide.md`

band 단위 파일을 권장하는 이유는 다음과 같다.

- 100개 전체를 한 파일에 넣는 것보다 검토 단위가 작다.
- 난이도 구간별 수정과 리뷰가 쉽다.
- 아트 테마나 보드 실루엣 변화도 band 기준으로 맞추기 좋다.

## 결론

100개 스테이지 확장은 `GDScript 하드코딩`이 아니라 `JSON 데이터 + Dictionary 런타임 + 검증 도구` 조합으로 가는 것이 가장 안전하다.

이 방식은 현재 구현을 크게 흔들지 않으면서도, 스테이지 추가와 난이도 조정, 저장 연동, 선택 화면 확장까지 이어지게 해 준다.
