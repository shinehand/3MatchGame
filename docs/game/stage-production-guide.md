# 스테이지 제작 가이드

## 1. 목적

- 이 문서는 새 스테이지를 어떤 방식으로 만들고 조정할지 정의한다.
- 개발자와 기획자가 같은 기준으로 `목표`, `이동 수`, `장애물`, `출현 확률`을 설계하도록 돕는다.

## 2. 현재 스테이지 데이터 구조

각 스테이지는 아래 항목을 한 세트로 가진다.

- `id`
- `name`
- `difficulty`
- `moves`
- `target_collect`
- `target_score`
- `target_blockers`
- `animal_pool`
- `spawn_weights`
- `blockers`
- `tutorial`

100스테이지 기준 추가 권장 필드:

- `board_mask`
- `band`
- `theme_key`
- `mechanics.enabled`
- `story_beat`
- `rescue_target`
- `boss_stage`

## 3. 스테이지 생성 기본 순서

1. 이번 스테이지가 가르칠 핵심을 하나 정한다.
2. 목표 타입을 정한다.
3. 이동 수를 정한다.
4. 블록 풀과 가중치를 정한다.
5. 장애물 위치와 수를 정한다.
6. 튜토리얼 문구를 짧게 작성한다.
7. 이번 스테이지의 구조 맥락을 한 줄로 작성한다.
8. 실제 플레이 후 난이도를 조정한다.

## 4. 스테이지별 학습 목표 예시

### Easy 구간

- 기본 스와이프 이해
- 목표 칩 읽기
- 수집 목표 우선순위 판단

### Normal 구간

- 복수 목표 처리
- 4매치 특수 블록 사용
- 점수 목표와 수집 목표 동시 관리

### Hard 구간

- 5매치 폭발 블록 활용
- 장애물과 점수의 동시 관리
- 복합 목표 플레이

## 5. 스토리 제작 규칙

### 기본 원칙

- 각 스테이지는 `무엇을 배우는가`와 함께 `누구를 구하는가`가 있어야 한다.
- 같은 밴드 안에서는 같은 정글 구역을 유지한다.
- 밴드 마지막 스테이지는 작은 피날레처럼 스토리 목적이 분명해야 한다.

### 밴드 단위 질문

1. 이번 10스테이지는 어떤 정글 구역인가
2. 이번 밴드에서 먼저 구하는 동물은 무엇인가
3. 밀렵꾼의 방해는 어떤 형태인가
4. 밴드 마지막에 어떤 진전을 보여 주는가

### 보스 스테이지 규칙

- 10, 20, 30... 100은 `미니 피날레` 또는 `보스성 구조 스테이지`로 취급한다.
- 목표는 많아질 수 있지만, 카피는 단순해야 한다.
- 예시: `코끼리 우리 앞의 동물들을 먼저 풀어 주세요`

## 6. 목표 타입 설계 규칙

### 수집 목표

- 특정 동물을 몇 개 모아야 하는지 정의
- 가장 기본적인 목표 타입

### 점수 목표

- 큰 매치와 연쇄를 유도할 때 사용
- 수집 목표만으로 부족할 때 긴장감을 만든다.

### 장애물 목표

- 레벨 개성을 만들기 위한 핵심 장치
- 중반 이후 난이도 상승 구간에서 적극 사용

### 복합 목표

- Hard 난이도에서만 사용
- 한 번에 너무 많은 목표를 넣지 않는다.

## 7. 이동 수 설계 규칙

- Easy: `12~13`
- Normal: `14~15`
- Hard: `15~16`

조정 기준:

- 블록 가중치가 유리하면 이동 수를 줄인다.
- 장애물이 많으면 이동 수를 소폭 늘린다.
- 점수 목표가 있으면 연쇄 여유를 위해 이동 수를 1 정도 더 준다.

## 8. 블록 풀과 가중치 설계

### 기본 원칙

- 쉬운 스테이지는 목표 동물이 더 자주 나오게 한다.
- 어려운 스테이지는 전체 가중치를 균등하게 맞춘다.
- 지나치게 한 종류만 몰아주면 퍼즐 다양성이 줄어든다.

### 예시

- Stage 1: 토끼 가중치 상승
- Stage 3: 곰/고양이 가중치 상승
- Hard: 전반적 균등 분포

## 9. 장애물 배치 규칙

### 현재 장애물: 덤불

- 덤불은 매치된 칸 또는 인접 칸의 충격으로 제거된다.
- 보드 중심 또는 십자형 구역에 두면 특수 블록 학습과 잘 맞는다.
- 스토리상으로는 덫, 시야 가림, 통로 방해물로 해석한다.

### 배치 원칙

- Easy에는 넣지 않는다.
- Normal부터 적은 수로 도입한다.
- Hard에서는 목표와 동선을 방해하도록 배치한다.
- 단, `막다른 보드`가 되지 않도록 한다.

## 10. 난이도 제작 방식

### 3단계 기준

- Easy: 규칙 소개
- Normal: 규칙 조합
- Hard: 규칙 압박

### 실제 제작 방법

1. Easy 2개 제작
2. Normal 2개 제작
3. Hard 2개 제작
4. 각 구간 마지막 스테이지는 다음 구간 학습 요소를 미리 암시

## 11. StageCatalog 운영 규칙

- 새 스테이지는 `stage_catalog.gd`에 데이터로 추가한다.
- 스테이지 로직은 최대한 데이터 기반으로 처리한다.
- 새 규칙을 만들 때도 먼저 데이터 필드 확장이 가능한지 확인한다.
- 100스테이지부터는 `scripts/stage_catalog.gd` 하드코딩보다 `JSON 카탈로그`를 원본으로 관리한다.
- 밴드 단위 파일 분리와 검증 도구를 같이 운영한다.

## 12. 플레이 테스트 기준

- 1판 평균 플레이 시간이 1~2분 내인지
- 첫 3수 안에 목표를 이해하는지
- 스테이지 시작 시 이번 구조 임무가 한 줄로 읽히는지
- 실패가 억울한지, 납득 가능한지
- 특수 블록이 최소 1회 이상 체감되는지
- 장애물이 단순 귀찮음이 아니라 전략 요소로 작동하는지

## 13. 실무 체크리스트

- 목표가 너무 많지 않은가
- 이동 수가 과도하게 빡빡하지 않은가
- 목표 동물 가중치가 적절한가
- 장애물 위치가 지나치게 막히지 않는가
- 튜토리얼 문구가 한 줄로 요약되는가
- 구조 맥락 문구가 규칙 설명보다 더 길지 않은가

## 14. 현재 초기 카탈로그 방향

- Stage 1: 토끼 수집
- Stage 2: 복수 수집 목표
- Stage 3: 4매치 특수 블록 도입
- Stage 4: 점수 + 장애물
- Stage 5: 폭발 블록 중심
- Stage 6: 복합 목표 하드 검증
